# Краткий аналитический отчёт

## Ускорение модели BERT для онлайн-классификации текстов

---

## 1. Выбранная модель

**BERT-base-uncased** — трансформерная модель с 110 млн параметров, применяемая для многоклассовой классификации текстовых обращений пользователей в системе технической поддержки.

| Характеристика | Значение |
|----------------|----------|
| Архитектура | Transformer Encoder (12 слоёв) |
| Параметры | 110 млн |
| Задача | Классификация текста (50 классов) |
| Режим работы | Онлайн-инференс |
| Текущее качество | Weighted F1 = 93.8% |

---

## 2. Целевая платформа и требования

### Аппаратная платформа
- **GPU:** NVIDIA T4 (16 GB VRAM, 65 TFLOPS FP16)
- **Bandwidth:** 320 GB/s

### Требования к производительности

| Метрика | Требование |
|---------|------------|
| Латентность p99 | ≤ 50 мс |
| Пропускная способность | ≥ 2000 RPS (пиковая) |
| Качество Weighted F1 | ≥ 91% |

---

## 3. Выявленные Bottlenecks

### Критические bottlenecks

| # | Bottleneck | Причина | Влияние |
|---|------------|---------|---------|
| 1 | **Memory bandwidth** | Малый batch size при онлайн-инференсе | Операции memory-bound, GPU недозагружен |
| 2 | **O(S²) сложность attention** | Архитектурная особенность self-attention | Резкий рост латентности с длиной последовательности |
| 3 | **Kernel launch overhead** | Много мелких операций | ~7% времени на накладные расходы при batch=1 |

### Дополнительные bottlenecks

| # | Bottleneck | Влияние |
|---|------------|---------|
| 4 | Softmax — memory-bound операция | Прерывает возможные fusion |
| 5 | Многократная загрузка весов | Не амортизируется при batch=1 |
| 6 | Недоиспользование Tensor Cores | 5-10% утилизации при batch=1 |

## 6. Инженерные компромиссы

### Сводка trade-offs

| Метод | Выигрыш | Цена |
|-------|---------|------|
| INT8 Quantization | 2× скорость, 4× память | 0.5-1% F1, калибровка |
| TensorRT | 1.5× скорость | Vendor lock-in, сложный debug |
| Dynamic Batching | 3× throughput | +latency variance |
| DistilBERT | 1.7× скорость, 40% меньше модель | 2-3% F1, недели на разработку |

### Ключевые риски

1. **INT8 drift** — необходима периодическая перекалибровка
2. **TensorRT версионирование** — engine привязан к GPU architecture
3. **Quality regression** — обязательный мониторинг метрик в production

---

## 7. Рекомендации

### Стратегия внедрения

1. **Неделя 1:** FP16 + TensorRT + Triton serving
2. **Неделя 2:** INT8 PTQ + Dynamic batching tuning
3. **Неделя 3:** Мониторинг, A/B тестирование, rollout

### Минимально жизнеспособная оптимизация

Для быстрого результата достаточно:
- **FP16 + TensorRT** — даёт 2× ускорение за 2-3 дня работы

### Полная оптимизация

Для максимального эффекта:
- **FP16 + TensorRT + INT8 + Batching** — даёт 5-6× ускорение за 2 недели

---

## 8. Заключение

Модель BERT-base для задачи классификации текстов может быть ускорена в **5-6 раз** при сохранении качества выше требуемого порога (Weighted F1 ≥ 91%). Основные bottlenecks связаны с memory-bound характером операций при малом batch size и квадратичной сложностью self-attention.

Рекомендуемый стек оптимизаций:
1. FP16 mixed precision (обязательно)
2. TensorRT компиляция (обязательно)
3. Dynamic batching через Triton (для throughput)
4. INT8 quantization (рекомендуется)

Ожидаемая экономия инфраструктурных затрат составляет **~$370,000/год** при первоначальных инвестициях ~$5,000 (2 недели инженерного времени).
