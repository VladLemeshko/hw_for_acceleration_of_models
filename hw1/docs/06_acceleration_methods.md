## 6.2 Квантизация

### 6.2.1 INT8 Post-Training Quantization (PTQ)

**Описание:** Преобразование весов и активаций модели из FP32/FP16 в INT8 формат после обучения.

**Затрагиваемые компоненты:**
- Все linear-слои (attention projections, FFN)
- Embedding layer (веса)
- Активации в процессе инференса

**Механизм ускорения:**
```
FP16 GEMM: C = A × B
  - Размер данных: 2 bytes/element
  - Tensor Core throughput: 65 TFLOPS

INT8 GEMM: C = scale × (A_int8 × B_int8) + bias
  - Размер данных: 1 byte/element
  - Tensor Core throughput: 130 TOPS (2× выше)
  - Память: 2× меньше bandwidth
```

**Ожидаемое ускорение:**
| Метрика | FP16 baseline | INT8 | Улучшение |
|---------|---------------|------|-----------|
| Latency (batch=1) | 15 мс | 8-10 мс | 1.5-2× |
| Throughput | 66 RPS | 100-130 RPS | 1.5-2× |
| Memory | 220 MB | 110 MB | 2× |

**Условия применимости:**

✅ **Применимо, когда:**
- Допустима деградация качества на 0.5-1%
- Есть репрезентативный калибровочный датасет
- Целевой hardware поддерживает INT8 (NVIDIA T4 ✓)

❌ **Не рекомендуется, когда:**
- Критична максимальная точность (medical, legal)
- Модель имеет outlier-активации (требуется специальная обработка)
- Hardware не имеет INT8 ускорителей

**Реализация:**
```python
import torch
from torch.quantization import quantize_dynamic

# Динамическая квантизация (простой вариант)
model_int8 = quantize_dynamic(
    model, 
    {torch.nn.Linear},
    dtype=torch.qint8
)

# Статическая квантизация с калибровкой (лучшее качество)
from pytorch_quantization import quant_modules
quant_modules.initialize()
# ... калибровка на representative dataset ...
```

### 6.2.2 FP16 Mixed Precision

**Описание:** Использование FP16 для вычислений с FP32 аккумуляцией для численной стабильности.

**Затрагиваемые компоненты:**
- Все GEMM операции
- Activation functions
- Attention computations

**Механизм ускорения:**
- Tensor Cores на T4 оптимизированы для FP16
- 2× меньше bandwidth для загрузки весов
- 2× больше данных в регистрах

**Ожидаемое ускорение:**
| Метрика | FP32 | FP16 | Улучшение |
|---------|------|------|-----------|
| Latency | 25 мс | 15 мс | 1.6× |
| Memory | 440 MB | 220 MB | 2× |

**Условия применимости:**

✅ **Применимо практически всегда** для inference на GPU с Tensor Cores

❌ **Осторожность при:**
- Очень глубоких моделях (>100 слоёв)
- Моделях с нестабильными градиентами (для training)

## 6.3 Дистилляция знаний (Knowledge Distillation)

### 6.3.1 DistilBERT

**Описание:** Обучение компактной модели (student) воспроизводить поведение исходной модели (teacher) с меньшим числом параметров.

**Архитектурные изменения:**

| Параметр | BERT-base | DistilBERT | Изменение |
|----------|-----------|------------|-----------|
| Layers | 12 | 6 | -50% |
| Hidden size | 768 | 768 | 0% |
| Attention heads | 12 | 12 | 0% |
| Parameters | 110M | 66M | -40% |

**Затрагиваемые компоненты:**
- Количество transformer-блоков (уменьшается вдвое)
- Все производные вычисления

**Механизм ускорения:**
```
BERT-base: 12 слоёв × (Attention + FFN + 2×LayerNorm)
DistilBERT: 6 слоёв × (Attention + FFN + 2×LayerNorm)

Теоретическое ускорение: ~2×
Практическое ускорение: 1.6-1.8× (overhead не масштабируется)
```

**Ожидаемое ускорение:**
| Метрика | BERT-base | DistilBERT | Улучшение |
|---------|-----------|------------|-----------|
| Latency | 15 мс | 8-9 мс | 1.7× |
| Parameters | 110M | 66M | 1.7× |
| Memory | 220 MB | 130 MB | 1.7× |
| F1 Score | 93.8% | 91-92% | -2-3% |

**Условия применимости:**

✅ **Применимо, когда:**
- Есть ресурсы на дообучение student-модели
- Допустима деградация качества на 2-3%
- Нужно значительное уменьшение размера модели

❌ **Не рекомендуется, когда:**
- Нет данных для дистилляции
- Критична максимальная точность
- Ограничено время на разработку

### 6.3.2 TinyBERT

**Описание:** Более агрессивная дистилляция с уменьшением hidden size.

| Параметр | BERT-base | TinyBERT-6L | TinyBERT-4L |
|----------|-----------|-------------|-------------|
| Layers | 12 | 6 | 4 |
| Hidden | 768 | 768 | 312 |
| Parameters | 110M | 66M | 14.5M |
| Speedup | 1× | 1.7× | 5-6× |
| F1 drop | 0% | 2-3% | 4-6% |

## 6.4 Pruning (Прореживание)

### 6.4.1 Structured Pruning (Attention Heads)

**Описание:** Удаление целых attention-голов, которые вносят минимальный вклад в качество.

**Затрагиваемые компоненты:**
- Multi-head attention в каждом слое
- Соответствующие проекционные матрицы

**Механизм:**
```
Original: 12 heads × 64 dim = 768 dim
Pruned (25%): 9 heads × 64 dim = 576 effective dim

Вычисления attention: пропорциональны числу голов
Memory: пропорциональна числу голов
```

**Ожидаемое ускорение:**
| Доля pruned голов | Speedup | F1 drop |
|-------------------|---------|---------|
| 25% (3 головы) | 1.15× | 0.5-1% |
| 50% (6 голов) | 1.3× | 1-2% |
| 75% (9 голов) | 1.5× | 3-5% |

**Условия применимости:**

✅ **Применимо, когда:**
- Модель overparameterized для задачи
- Есть возможность fine-tuning после pruning
- Нужно умеренное ускорение с минимальной деградацией

❌ **Не рекомендуется, когда:**
- Нет ресурсов на анализ важности голов
- Задача требует всех типов attention-паттернов

### 6.4.2 Unstructured Pruning

**Описание:** Обнуление отдельных весов по критерию magnitude.

**Ограничения:**
- Требует sparse hardware/software support
- NVIDIA T4 имеет ограниченную поддержку sparsity
- Без специализированной поддержки не даёт ускорения

**Вывод:** Для данного сценария unstructured pruning **не рекомендуется**.

## 6.5 Системные оптимизации

### 6.5.1 TensorRT Optimization

**Описание:** Компиляция модели в оптимизированный runtime с применением graph-level оптимизаций.

**Применяемые оптимизации:**

| Оптимизация | Описание | Эффект |
|-------------|----------|--------|
| **Layer Fusion** | Объединение последовательных операций | Меньше kernel launches |
| **Kernel Auto-tuning** | Выбор оптимальных CUDA kernels | Максимальная производительность |
| **Precision Calibration** | Автоматический выбор точности | INT8/FP16 без потери качества |
| **Memory Optimization** | Оптимизация использования памяти | Меньше memory traffic |

**Пример fusion:**
```
До fusion:
  GEMM → Bias Add → LayerNorm → GELU → GEMM → ...
  (5 kernel launches)

После fusion:
  Fused_GEMM_Bias_LayerNorm_GELU_GEMM
  (1 kernel launch)
```

**Ожидаемое ускорение:**
| Метрика | PyTorch | TensorRT FP16 | TensorRT INT8 |
|---------|---------|---------------|---------------|
| Latency (batch=1) | 15 мс | 8-10 мс | 5-7 мс |
| Latency (batch=32) | 55 мс | 30-35 мс | 20-25 мс |
| Speedup | 1× | 1.5-2× | 2-3× |

**Условия применимости:**

✅ **Рекомендуется для production deployment на NVIDIA GPU**

❌ **Ограничения:**
- Время компиляции: 5-30 минут
- Фиксированные input shapes (или dynamic shapes с оверхедом)
- Vendor lock-in на NVIDIA

### 6.5.2 CUDA Graphs

**Описание:** Захват последовательности CUDA операций в граф для минимизации CPU overhead.

**Механизм:**
```
Обычное выполнение:
  CPU: launch_kernel_1 → launch_kernel_2 → ... → launch_kernel_N
  GPU:        kernel_1 →        kernel_2 → ... →        kernel_N
  
  Overhead: N × ~5 мкс = значительный при малых kernels

CUDA Graph:
  CPU: launch_graph (один раз)
  GPU: все kernels выполняются по записанному плану
  
  Overhead: 1 × ~10 мкс = минимальный
```

**Ожидаемое ускорение:**
| Batch size | Без CUDA Graphs | С CUDA Graphs | Улучшение |
|------------|-----------------|---------------|-----------|
| 1 | 15 мс | 13-14 мс | 7-10% |
| 8 | 25 мс | 24 мс | 3-5% |
| 32 | 55 мс | 54 мс | <2% |

**Условия применимости:**

✅ **Эффективно при:**
- Малых batch sizes
- Фиксированных input shapes
- Многократных инференсах с одинаковой структурой

❌ **Ограничения:**
- Не поддерживает dynamic shapes
- Сложнее debug
- Ограниченная совместимость с некоторыми операциями

### 6.5.3 Dynamic Batching

**Описание:** Группировка поступающих запросов в батчи для повышения throughput.

**Механизм:**
```
Без батчинга: каждый запрос обрабатывается отдельно
  Request 1 → [GPU] → Response 1
  Request 2 → [GPU] → Response 2
  ...
  Throughput: 66 RPS (при 15 мс/запрос)

С батчингом: запросы группируются
  [Request 1, 2, ..., 8] → [GPU] → [Response 1, 2, ..., 8]
  Latency: 25 мс (для batch=8)
  Throughput: 8 × 1000/25 = 320 RPS
```

**Trade-off latency vs throughput:**

| Batch size | Latency | Throughput | Efficiency |
|------------|---------|------------|------------|
| 1 | 15 мс | 66 RPS | 0.1% GPU util |
| 4 | 18 мс | 220 RPS | 30% |
| 8 | 25 мс | 320 RPS | 45% |
| 16 | 35 мс | 450 RPS | 60% |
| 32 | 55 мс | 580 RPS | 75% |

**Реализация с NVIDIA Triton:**
```python
# config.pbtxt
dynamic_batching {
  preferred_batch_size: [4, 8, 16]
  max_queue_delay_microseconds: 10000  # 10 мс ожидание
}
```

**Условия применимости:**

✅ **Эффективно при:**
- Высокой частоте запросов
- Допустимости небольшого увеличения latency
- Приоритете throughput над latency

❌ **Ограничения:**
- Увеличивает latency для отдельных запросов
- Требует tuning параметров
- Может быть неэффективен при нестабильной нагрузке

### 6.5.4 Kernel Fusion (FlashAttention)

**Описание:** Оптимизированная реализация attention, минимизирующая обращения к глобальной памяти.

**Проблема стандартного attention:**
```
Q, K, V: (B, H, S, D) в HBM

1. S = Q @ K.T          # Read Q, K; Write S to HBM
2. P = softmax(S)       # Read S; Write P to HBM  
3. O = P @ V            # Read P, V; Write O to HBM

Total HBM access: O(B × H × S² × D) — квадратичная по S
```

**FlashAttention:**
```
Tiled computation:
- Загрузка блоков Q, K, V в SRAM
- Вычисление attention внутри блока
- Запись только финального результата

Total HBM access: O(B × H × S × D) — линейная по S
```

**Ожидаемое ускорение:**

| Seq length | Standard Attention | FlashAttention | Speedup |
|------------|-------------------|----------------|---------|
| 128 | 5 мс | 3 мс | 1.7× |
| 256 | 18 мс | 6 мс | 3× |
| 512 | 70 мс | 15 мс | 4.5× |

**Условия применимости:**

✅ **Рекомендуется для:**
- Длинных последовательностей (S > 256)
- Memory-bound сценариев
- GPU с достаточным SRAM (T4 ✓)

❌ **Ограничения:**
- Требует специфической интеграции
- Не все attention variants поддерживаются
- Overhead при коротких последовательностях

## 6.6 Архитектурные модификации

### 6.6.1 Early Exit (Adaptive Inference)

**Описание:** Добавление промежуточных классификаторов для "простых" примеров.

**Механизм:**
```
┌─────────────┐
│   Input     │
└──────┬──────┘
       ↓
┌──────────────┐
│  Layers 1-3  │──→ Exit 1 (confidence > 0.95?) ──→ Output
└──────┬───────┘
       ↓
┌──────────────┐
│  Layers 4-6  │──→ Exit 2 (confidence > 0.90?) ──→ Output
└──────┬───────┘
       ↓
┌──────────────┐
│  Layers 7-12 │──→ Final Output
└──────────────┘
```

**Ожидаемое ускорение:**
- Если 50% примеров выходят после 6 слоёв: ~1.5× средний speedup
- Сильно зависит от распределения сложности данных

**Условия применимости:**

✅ **Эффективно при:**
- Неравномерной сложности примеров
- Допустимости вариативной latency

❌ **Не рекомендуется:**
- При строгих требованиях к стабильности latency
- При равномерно сложных данных

## 6.8 Рекомендуемая стратегия оптимизации

Для рассматриваемого сценария (онлайн-инференс, p99 latency ≤50 мс, throughput ≥2000 RPS) рекомендуется следующая последовательность оптимизаций:

### Этап 1: Quick Wins (1-2 дня)
1. **FP16 inference** — нет потери качества, 1.6× speedup
2. **TensorRT конвертация** — дополнительно 1.3× speedup
3. **Dynamic batching** через Triton — увеличение throughput

**Ожидаемый результат:** Latency ~8 мс, Throughput ~300 RPS на 1 GPU

### Этап 2: Квантизация (3-5 дней)
4. **INT8 PTQ** с калибровкой — ещё 1.5× speedup
5. Валидация качества на тестовом наборе

**Ожидаемый результат:** Latency ~5-6 мс, Throughput ~500 RPS на 1 GPU

### Этап 3: Архитектурные изменения (1-2 недели, если необходимо)
6. **Дистилляция в DistilBERT** — если нужно дальнейшее ускорение
7. **FlashAttention** интеграция — для длинных последовательностей

**Ожидаемый результат:** Latency ~3-4 мс, качество F1 ≥91%
