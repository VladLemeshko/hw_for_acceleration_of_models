# 4. Вычислительные затраты и узкие места

## 4.1 Методология анализа

Для выявления bottlenecks модели BERT на платформе NVIDIA T4 используется комплексный подход, включающий:

1. **Теоретический анализ** — оценка вычислительной сложности операций
2. **Roofline-модель** — определение, ограничены ли операции вычислениями или памятью
3. **Профилирование** — измерение фактического времени выполнения операций

## 4.2 Roofline-анализ

### Модель производительности

Roofline-модель связывает производительность операции с её **арифметической интенсивностью** (Arithmetic Intensity, AI):

$$AI = \frac{\text{FLOPs}}{\text{Bytes transferred}}$$

$$\text{Performance} = \min(\text{Peak FLOPS}, \text{AI} \times \text{Memory Bandwidth})$$

### Параметры платформы NVIDIA T4

| Метрика | Значение |
|---------|----------|
| Пиковая производительность FP16 | 65 TFLOPS |
| Пиковая производительность INT8 | 130 TOPS |
| Пропускная способность памяти | 320 GB/s |
| **Ridge Point (FP16)** | 65 TFLOPS / 320 GB/s = **203 FLOP/Byte** |

Операции с AI < 203 FLOP/Byte являются **memory-bound** (ограничены пропускной способностью памяти).

### Арифметическая интенсивность операций BERT

| Операция | AI (FLOP/Byte) | Тип ограничения |
|----------|----------------|-----------------|
| Embedding lookup | ~0.5 | Memory-bound |
| LayerNorm | ~10 | Memory-bound |
| Softmax | ~15 | Memory-bound |
| GELU activation | ~20 | Memory-bound |
| Attention QKV (batch=1) | ~85 | Memory-bound |
| Attention QKV (batch=32) | ~680 | **Compute-bound** |
| FFN (batch=1) | ~170 | Memory-bound |
| FFN (batch=32) | ~1360 | **Compute-bound** |

**Ключевой вывод:** При малых batch size (характерных для онлайн-инференса) большинство операций являются memory-bound, что ограничивает достижимую производительность.

## 4.3 Детальный анализ операций

### 4.3.1 Матричные умножения (GEMM)

Матричные умножения составляют основу вычислений в BERT:

**В Attention-блоке:**
- QKV projection: 3 × (B × S × H) @ (H × H) = 3 GEMM
- Output projection: (B × S × H) @ (H × H) = 1 GEMM
- Score computation: (B × A × S × D) @ (B × A × D × S) = batched GEMM
- Context computation: (B × A × S × S) @ (B × A × S × D) = batched GEMM

**В FFN-блоке:**
- Up-projection: (B × S × H) @ (H × 4H) = 1 GEMM
- Down-projection: (B × S × 4H) @ (4H × H) = 1 GEMM

**Анализ эффективности:**

Для GEMM C = A × B размером (M × K) @ (K × N):
- FLOPs = 2 × M × K × N
- Memory = (M × K + K × N + M × N) × sizeof(dtype)
- AI = 2MKN / [(MK + KN + MN) × bytes]

При batch=1, seq_len=128, hidden=768:
```
QKV projection: M=128, K=768, N=768
FLOPs = 2 × 128 × 768 × 768 = 151 MFLOPs
Memory (FP16) = (128×768 + 768×768 + 128×768) × 2 = 1.38 MB
AI = 151 / 1.38 ≈ 109 FLOP/Byte — memory-bound!
```

При batch=32:
```
M = 32 × 128 = 4096
AI = 2 × 4096 × 768 × 768 / [(4096×768 + 768×768 + 4096×768) × 2]
   ≈ 680 FLOP/Byte — compute-bound
```

### 4.3.2 Self-Attention

Self-attention включает несколько подоперации с различными характеристиками:

**Score computation: QK^T**
```
Размеры: (B × A × S × D) @ (B × A × D × S) → (B × A × S × S)
При S=128, A=12, D=64:
FLOPs = B × A × 2 × S × D × S = B × 12 × 2 × 128 × 64 × 128 = B × 25.2 MFLOPs
Memory = B × A × (S×D + D×S + S×S) × 2 = B × 12 × (8K + 8K + 16K) × 2 = B × 0.75 MB
AI ≈ 33 FLOP/Byte — memory-bound
```

**Softmax**

Softmax — элементная операция с низкой арифметической интенсивностью:
```
Для матрицы S×S нужно:
- Найти max по каждой строке: S операций на строку
- Вычесть max и экспоненту: S операций
- Сумма и деление: S операций
Итого: ~4S операций на строку, S строк → 4S² операций
Memory: 2 × S² × sizeof(dtype) (чтение + запись)
AI ≈ 4S² / (4 × S²) = 1 FLOP/Byte для FP32, 2 для FP16
```

Softmax является **крайне memory-bound** операцией.

### 4.3.3 Layer Normalization

LayerNorm вычисляет статистики по скрытому измерению:

```
Для тензора (B × S × H):
- Вычисление mean: H сложений на позицию
- Вычисление variance: H операций
- Нормализация: H операций (вычитание, деление)
- Scale и shift: 2H операций
Итого: ~5H операций на позицию, B×S позиций
FLOPs ≈ 5 × B × S × H
Memory = 2 × B × S × H × 2 (чтение входа + запись выхода, FP16)
AI ≈ 5H / 4H = 1.25 FLOP/Byte
```

LayerNorm — **сильно memory-bound** операция.

### 4.3.4 Активации (GELU)

GELU — поэлементная операция:
```
GELU(x) = x × Φ(x) ≈ 0.5x(1 + tanh(√(2/π)(x + 0.044715x³)))
```

При аппроксимации требуется ~20 операций на элемент:
```
FLOPs ≈ 20 × B × S × 4H (применяется после up-projection)
Memory = 2 × B × S × 4H × 2
AI ≈ 20 / 4 = 5 FLOP/Byte
```

GELU — **memory-bound** операция.

## 4.4 Профиль затрат времени

### Теоретическое распределение (batch=1, seq_len=128)

| Операция | Время (мс) | Доля | Тип ограничения |
|----------|-----------|------|-----------------|
| Attention GEMM (QKV+Out) | 3.2 | 22% | Memory |
| Attention Score+Softmax+Context | 2.8 | 19% | Memory |
| FFN GEMM | 4.5 | 31% | Memory |
| LayerNorm (×24) | 1.8 | 12% | Memory |
| GELU (×12) | 0.9 | 6% | Memory |
| Residual add | 0.5 | 3% | Memory |
| Kernel launch overhead | 1.0 | 7% | Latency |
| **Итого** | **~14.7** | **100%** | — |

### Изменение профиля при увеличении batch size

| Batch size | Attention | FFN | Kernel overhead | Общее время |
|------------|-----------|-----|-----------------|-------------|
| 1 | 41% | 37% | 7% | ~15 мс |
| 8 | 43% | 42% | 2% | ~25 мс |
| 32 | 45% | 47% | <1% | ~55 мс |
| 64 | 46% | 48% | <1% | ~100 мс |

При росте batch size доля накладных расходов снижается, а операции становятся compute-bound.

## 4.5 Идентифицированные Bottlenecks

### Bottleneck #1: Memory Bandwidth при малом batch size

**Описание:** При batch_size = 1 (типичный сценарий онлайн-инференса) большинство операций ограничены пропускной способностью памяти, а не вычислительной мощностью GPU.

**Причина:** Низкая арифметическая интенсивность операций из-за малого объёма данных не позволяет "замаскировать" латентность доступа к памяти.

**Количественная оценка:**
- Теоретическая производительность GEMM при batch=1: ~50 GFLOPS (0.08% от пика FP16)
- При batch=32: ~30 TFLOPS (46% от пика)

**Проявление:** Линейное масштабирование времени инференса с batch size в диапазоне 1-16, затем суб-линейное.

### Bottleneck #2: Квадратичная сложность Self-Attention

**Описание:** Вычислительная и memory сложность self-attention масштабируется как O(S²), где S — длина последовательности.

**Причина:** Attention-механизм требует вычисления попарных взаимодействий между всеми токенами.

**Количественная оценка:**
```
При S=64:   Attention FLOPs = 100 MFLOPs, Memory = 0.2 MB
При S=128:  Attention FLOPs = 400 MFLOPs, Memory = 0.8 MB
При S=256:  Attention FLOPs = 1.6 GFLOPs, Memory = 3.2 MB
При S=512:  Attention FLOPs = 6.4 GFLOPs, Memory = 12.8 MB
```

**Проявление:** Резкое увеличение латентности при обработке длинных последовательностей.

### Bottleneck #3: Накладные расходы на запуск Kernel'ов

**Описание:** При малом batch size значительная часть времени тратится на запуск CUDA kernel'ов, а не на вычисления.

**Причина:** Каждая операция в BERT (GEMM, softmax, LayerNorm и т.д.) требует отдельного kernel launch с накладными расходами ~5-10 мкс.

**Количественная оценка:**
```
BERT-base содержит примерно 150-200 kernel вызовов на forward pass
Overhead: 200 × 5 мкс = 1 мс
При общем времени инференса 15 мс это составляет ~7%
```

**Проявление:** Непропорциональное увеличение латентности при уменьшении batch size.

### Bottleneck #4: Неэффективность Softmax

**Описание:** Softmax является memory-bound операцией, которая разбивает потенциально fused цепочку вычислений.

**Причина:** 
- Softmax требует глобальной синхронизации (max и sum по всей строке)
- Низкая арифметическая интенсивность (~2 FLOP/Byte)
- Невозможность слияния с предшествующим GEMM из-за зависимости по данным

**Количественная оценка:**
- 12 слоёв × 12 голов = 144 вызова softmax на forward pass
- Каждый вызов: ~50 мкс при S=128
- Суммарно: ~7 мс (до 50% времени attention)

### Bottleneck #5: Многократная загрузка весов

**Описание:** При batch_size = 1 веса модели загружаются из глобальной памяти GPU многократно, не амортизируясь по примерам в батче.

**Причина:** Веса BERT (~440 MB в FP32, ~220 MB в FP16) не помещаются в кэши GPU и загружаются из GDDR6 при каждом использовании.

**Количественная оценка:**
```
Загрузка весов FFN одного слоя: 768×3072×2 + 3072×768×2 = 9.4 MB (FP16)
При bandwidth 320 GB/s: 9.4 MB / 320 GB/s = 29 мкс (только чтение)
12 слоёв: 29 × 12 × 2 = 0.7 мс
Реальное время больше из-за неоптимального паттерна доступа
```

### Bottleneck #6: Низкая утилизация Tensor Cores

**Описание:** Tensor Cores, обеспечивающие максимальную производительность на T4, недоиспользуются при малых batch size.

**Причина:**
- Tensor Cores эффективны при крупных GEMM с выровненными размерностями
- При batch=1 матрицы слишком малы для полной загрузки Tensor Cores
- Часть операций (softmax, LayerNorm, GELU) не используют Tensor Cores

**Количественная оценка:**
- Реальная утилизация Tensor Cores при batch=1: ~5-10%
- При batch=32: ~40-60%
- При batch=64: ~60-80%

## 4.7 Резюме

| Bottleneck | Критичность | Затрагиваемый ресурс | Основная причина |
|------------|-------------|---------------------|------------------|
| Memory bandwidth | Критическая | Время | Малый batch size |
| O(S²) attention | Высокая | Время, память | Архитектура |
| Kernel overhead | Средняя | Время | Много мелких операций |
| Softmax inefficiency | Средняя | Время | Memory-bound операция |
| Weight reloading | Средняя | Bandwidth | Размер модели |
| Tensor Core underuse | Высокая | Время | Малый batch size |

Эти bottlenecks определяют направления оптимизации, рассмотренные в следующем разделе.
