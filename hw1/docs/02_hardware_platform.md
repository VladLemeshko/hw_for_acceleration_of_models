# 2. Целевая аппаратная платформа

## 2.1 Выбор платформы

Для развёртывания модели выбрана **серверная GPU-платформа** на базе ускорителя **NVIDIA T4**. Данный выбор обусловлен балансом между вычислительной мощностью, энергоэффективностью и стоимостью владения, что критично для production-сценариев с высокой нагрузкой.

## 2.3 Ключевые ограничения платформы

### 2.3.1 Ограничения по памяти GPU

**Доступная память: 16 GB GDDR6**

Память GPU является критическим ресурсом, который необходимо эффективно распределять между:

1. **Веса модели**: BERT-base в FP32 занимает ~440 MB, в FP16 — ~220 MB
2. **Активации**: Зависят от batch size и длины последовательности
3. **KV-cache**: Для трансформерных моделей требуется хранение промежуточных состояний
4. **Накладные расходы фреймворка**: CUDA runtime, cuDNN, служебные буферы (~1-2 GB)

**Расчёт потребления памяти для инференса:**

```
Память_модели (FP16) ≈ 220 MB
Память_активаций ≈ batch_size × seq_len × hidden_dim × num_layers × 2 bytes
                 ≈ 32 × 128 × 768 × 12 × 2 ≈ 75 MB
Служебная память ≈ 2 GB
───────────────────────────────
Итого при batch=32: ≈ 2.3 GB
```

При указанных параметрах память не является критическим ограничением, однако при увеличении batch size или длины последовательности потребление растёт линейно.

### 2.3.2 Ограничения пропускной способности памяти

**Bandwidth: 320 GB/s**

Пропускная способность памяти часто становится bottleneck для операций с низкой арифметической интенсивностью. Для трансформерных моделей критичны:

- **Загрузка весов**: При малых batch size время загрузки весов превышает время вычислений
- **Attention-операции**: Чтение/запись матриц Q, K, V требует значительного bandwidth
- **Softmax и нормализации**: Операции с низкой интенсивностью вычислений

**Арифметическая интенсивность BERT:**

Для операции матричного умножения C = A × B:
- Вычисления: 2 × M × N × K FLOP
- Передача данных: (M × K + K × N + M × N) × bytes

При batch_size = 1 арифметическая интенсивность низкая, и производительность ограничивается bandwidth.

### 2.3.3 Ограничения по латентности

**Требуемая латентность: ≤ 50 мс (p99)**

Латентность инференса складывается из нескольких компонентов:

| Компонент | Типичное время | Вклад в общую латентность |
|-----------|----------------|---------------------------|
| Сетевой ввод-вывод | 1-5 мс | 5-10% |
| Препроцессинг (токенизация) | 0.5-2 мс | 2-5% |
| Передача данных CPU→GPU | 0.1-0.5 мс | ~1% |
| Вычисления на GPU | 5-30 мс | 60-80% |
| Передача результата GPU→CPU | 0.05-0.1 мс | <1% |
| Постпроцессинг | 0.1-0.5 мс | ~1% |
| Накладные расходы фреймворка | 2-5 мс | 10-15% |

**Критичность:** При обработке одиночных запросов (batch_size = 1) накладные расходы на запуск kernel'ов и синхронизацию становятся значительной частью общей латентности.

### 2.3.4 Вычислительные ограничения

**FP16 Tensor Cores: 65 TFLOPS**

Tensor Cores обеспечивают максимальную производительность при выполнении операций:
- Матричного умножения в формате FP16
- Mixed precision (FP16 вычисления, FP32 аккумуляция)
- INT8 квантизованных операций

**Ограничения Tensor Cores:**
- Требуют выравнивания размерностей матриц (кратность 8 для FP16, 16 для INT8)
- Максимальная эффективность при крупных batch size
- Не все операции могут использовать Tensor Cores (только GEMM-подобные)

### 2.3.5 Энергопотребление

**TDP: 70 Вт**

NVIDIA T4 спроектирован как энергоэффективный ускоритель для инференса:
- Низкое энергопотребление позволяет размещение в стандартных серверных стойках
- Пассивное охлаждение (требуется хороший воздушный поток в серверном шасси)
- Соотношение производительность/ватт выше, чем у training-ориентированных GPU

**Для данного сценария** энергопотребление не является критическим ограничением, поскольку инфраструктурные затраты определяются в первую очередь стоимостью GPU-часов в облаке.

### Совместимость с оптимизациями

NVIDIA T4 поддерживает следующие техники оптимизации:
- ✓ FP16 mixed precision (нативная поддержка Tensor Cores)
- ✓ INT8 квантизация (требуется калибровка)
- ✓ TensorRT оптимизации (layer fusion, kernel auto-tuning)
- ✓ CUDA Graphs (снижение накладных расходов на запуск kernel'ов)
- ✓ Multi-stream execution (параллельное выполнение независимых запросов)
