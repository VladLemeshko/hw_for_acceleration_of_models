# 7. Инженерные компромиссы

## 7.1 Введение

Любая оптимизация производительности модели машинного обучения сопряжена с компромиссами. Данный раздел систематизирует ожидаемые trade-offs для предложенных методов ускорения, позволяя принять обоснованное решение о применимости каждого метода.

## 7.3 Компромиссы по методам оптимизации

### 7.3.1 FP16 Mixed Precision

| Выигрыш | Цена |
|---------|------|
| ✅ 1.6× ускорение | ⚠️ Минимальные: редкие edge-cases с overflow |
| ✅ 2× экономия памяти | ⚠️ Необходимость проверки численной стабильности |
| ✅ Нет потери качества | |

**Детальный анализ:**

**Что выигрываем:**
- Скорость: Tensor Cores работают на FP16 в 2× быстрее
- Память: веса занимают 220 MB вместо 440 MB
- Bandwidth: 2× меньше данных передаётся из памяти

**Чем платим:**
- Риск overflow при экстремальных значениях активаций
- Необходимость тестирования на edge-cases
- Минимальные изменения в коде (torch.autocast)

**Рекомендация:** **Безусловно применять.** Соотношение benefit/cost исключительно выгодное.

### 7.3.2 INT8 Quantization

| Выигрыш | Цена |
|---------|------|
| ✅ 2× ускорение | ⚠️ 0.5-1% потеря F1 |
| ✅ 4× экономия памяти (vs FP32) | ⚠️ Необходимость калибровки |
| ✅ Использование INT8 Tensor Cores | ⚠️ Время на отладку |

**Детальный анализ:**

**Что выигрываем:**
- Скорость: INT8 Tensor Cores дают 130 TOPS vs 65 TFLOPS FP16
- Память: 110 MB вместо 220 MB (FP16)
- Throughput: 2× больше запросов в секунду

**Чем платим:**

1. **Качество модели:**
   ```
   Weighted F1: 93.8% → 93.0-93.3%
   Accuracy: 94.2% → 93.5-94.0%
   
   Практический эффект: 
   - ~50-100 дополнительных ошибок в день (из 10,000 обращений)
   - Приемлемо для большинства сценариев
   ```

2. **Затраты на разработку:**
   - Калибровка: 2-4 часа на подготовку representative dataset
   - Валидация: 1-2 дня на тестирование качества
   - Отладка: возможны проблемы с outlier-активациями

3. **Операционная сложность:**
   - Отдельный pipeline для квантизации
   - Периодическая перекалибровка при drift данных

**Матрица решения:**

| Условие | Рекомендация |
|---------|--------------|
| Критична максимальная точность | Не применять |
| Допустима деградация 0.5-1% | Применять PTQ |
| Нужно максимальное ускорение | Применять с fine-tuning (QAT) |

### 7.3.3 Knowledge Distillation (DistilBERT)

| Выигрыш | Цена |
|---------|------|
| ✅ 1.7× ускорение | ⚠️ 2-3% потеря F1 |
| ✅ 40% меньше параметров | ⚠️ Недели на обучение student |
| ✅ Пропорционально меньше памяти | ⚠️ Требуется GPU для обучения |

**Детальный анализ:**

**Что выигрываем:**
- Фундаментально меньшая модель (6 слоёв vs 12)
- Постоянное ускорение, комбинируемое с другими методами
- Меньший carbon footprint в production

**Чем платим:**

1. **Качество модели:**
   ```
   BERT-base Weighted F1: 93.8%
   DistilBERT Weighted F1: 91.0-92.0%
   
   Практический эффект:
   - 200-300 дополнительных ошибок в день
   - Может быть критично для некоторых категорий
   ```

2. **Затраты на разработку:**
   - Подготовка distillation pipeline: 3-5 дней
   - Обучение student: 1-2 дня на 8× V100
   - Hyperparameter tuning: 2-3 дня
   - Итого: **1-2 недели** инженерного времени

3. **Инфраструктурные затраты:**
   ```
   Обучение DistilBERT:
   - 8× V100 × 24 часа × $3/hour = ~$600
   - Можно использовать pretrained DistilBERT (бесплатно)
   - Fine-tuning на своих данных: 8× V100 × 4 часа = ~$100
   ```

4. **Технический долг:**
   - Два репозитория моделей (teacher и student)
   - Необходимость синхронизации при изменениях
   - Сложнее A/B тестирование

**Матрица решения:**

| Условие | Рекомендация |
|---------|--------------|
| Другие методы достаточны | Не применять |
| Нужно значительное ускорение | Применять с готовым DistilBERT |
| Критично качество | Не применять |

### 7.3.4 TensorRT Optimization

| Выигрыш | Цена |
|---------|------|
| ✅ 1.5-2× ускорение | ⚠️ Vendor lock-in (NVIDIA) |
| ✅ Автоматические оптимизации | ⚠️ Время компиляции (10-30 мин) |
| ✅ Нет потери качества | ⚠️ Сложнее debugging |

**Детальный анализ:**

**Что выигрываем:**
- Layer fusion: меньше kernel launches
- Auto-tuning: оптимальные CUDA kernels
- Memory optimization: оптимальное использование памяти
- Комбинируется с FP16/INT8

**Чем платим:**

1. **Vendor lock-in:**
   - Модель работает только на NVIDIA GPU
   - Невозможно перенести на AMD, Intel, или CPU
   - Зависимость от TensorRT версий

2. **Операционная сложность:**
   ```
   Deployment pipeline:
   
   PyTorch модель
        ↓ (export to ONNX)
   ONNX модель  
        ↓ (TensorRT conversion, 10-30 мин)
   TensorRT engine (platform-specific!)
        ↓
   Deployment
   
   Проблемы:
   - Engine специфичен для GPU architecture
   - Перекомпиляция при смене hardware
   - Версионная совместимость TensorRT/CUDA/cuDNN
   ```

3. **Debugging сложность:**
   - TensorRT — чёрный ящик
   - Сложно диагностировать численные проблемы
   - Ограниченные возможности профилирования

4. **Поддержка операторов:**
   - Не все PyTorch операции поддерживаются
   - Custom operators требуют plugins
   - Динамические формы работают медленнее

**Матрица решения:**

| Условие | Рекомендация |
|---------|--------------|
| Deployment на NVIDIA GPU | Применять |
| Нужна портативность | Не применять |
| Часто меняется архитектура модели | Осторожно (долгая перекомпиляция) |

### 7.3.5 Dynamic Batching

| Выигрыш | Цена |
|---------|------|
| ✅ 2-5× throughput | ⚠️ Увеличение latency |
| ✅ Лучшая GPU utilization | ⚠️ Сложность конфигурации |
| ✅ Нет потери качества | ⚠️ Вариативность latency |

**Детальный анализ:**

**Что выигрываем:**
- GPU не простаивает между запросами
- Амортизация накладных расходов на batch
- Экономия на количестве GPU

**Чем платим:**

1. **Увеличение latency:**
   ```
   Без батчинга: p99 latency = 15 мс
   С батчингом (wait 10 мс): p99 latency = 25-30 мс
   
   Trade-off curve:
   
   Latency (мс)
   50 │                           *
   40 │                      *
   30 │                 *
   20 │            *
   15 │       *
   10 │  *
      └─────────────────────────────────
         0   100  200  300  400  500 RPS (throughput)
   ```

2. **Вариативность latency:**
   - Запрос, попавший в начало batch window, ждёт дольше
   - p99 - p50 gap увеличивается
   - Сложнее гарантировать SLA

3. **Сложность tuning:**
   - max_batch_size: больше — выше throughput, выше latency
   - max_queue_delay: компромисс между ожиданием и эффективностью
   - preferred_batch_sizes: влияет на GPU utilization

**Параметры для сценария:**

```
Требования: p99 ≤ 50 мс, throughput ≥ 2000 RPS

Рекомендуемые настройки:
- max_batch_size: 16
- max_queue_delay: 10 мс
- preferred_batch_sizes: [4, 8, 16]

Ожидаемый результат:
- p99 latency: ~35-40 мс (с TensorRT INT8)
- Throughput: ~500 RPS на 1 GPU
- Для 2000 RPS: 4 GPU
```

### 7.3.6 Structured Pruning

| Выигрыш | Цена |
|---------|------|
| ✅ 1.15-1.3× ускорение | ⚠️ 0.5-2% потеря качества |
| ✅ Меньше параметров | ⚠️ Требуется анализ важности |
| ✅ Совместимо со standard hardware | ⚠️ Fine-tuning после pruning |

**Детальный анализ:**

**Что выигрываем:**
- Удаление redundant attention heads
- Пропорциональное ускорение attention
- Меньший model footprint

**Чем платим:**

1. **Качество:**
   ```
   Pruning 25% heads:
   - F1: 93.8% → 93.0-93.5%
   - Некоторые паттерны attention теряются
   
   Pruning 50% heads:
   - F1: 93.8% → 91.5-92.5%
   - Значительная деградация на сложных примерах
   ```

2. **Затраты на разработку:**
   - Анализ важности голов: 1-2 дня
   - Fine-tuning после pruning: 0.5-1 день
   - Валидация: 0.5 дня
   - Итого: **2-4 дня**

3. **Ограниченный эффект:**
   - Ускоряет только attention (не FFN)
   - При 50% pruning голов — только 15% overall speedup
   - ROI ниже, чем у quantization

**Рекомендация:** Применять как дополнительную оптимизацию, если другие методы исчерпаны.

## 7.5 Анализ рисков

### 7.5.1 Риски потери качества

| Метод | Риск | Митигация |
|-------|------|-----------|
| INT8 | Outlier-активации могут деградировать сильнее | Per-channel quantization |
| DistilBERT | Сложные категории страдают больше | Ensemble с teacher для hard cases |
| Pruning | Удаление важных heads | Importance analysis перед pruning |

### 7.5.2 Операционные риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| TensorRT несовместимость | Средняя | Высокое | Fallback на PyTorch |
| INT8 drift при обновлении данных | Средняя | Среднее | Periodic recalibration |
| Версионные конфликты | Высокая | Среднее | Strict version pinning |
| Debugging production issues | Средняя | Высокое | Comprehensive logging |

### 7.5.3 Организационные риски

| Риск | Митигация |
|------|-----------|
| Knowledge silos (только один человек понимает оптимизации) | Документация, pair programming |
| Сложность onboarding новых инженеров | Runbooks, training |
| Vendor dependency | Оценка альтернатив (ONNX Runtime, OpenVINO) |

## 7.6 Total Cost of Ownership (TCO)

### Сравнение вариантов для обслуживания 2000 RPS

| Конфигурация | GPU | $/month | Dev effort | Quality |
|--------------|-----|---------|------------|---------|
| Baseline PyTorch FP32 | 60 T4 | $36,000 | 0 | 93.8% F1 |
| + FP16 | 36 T4 | $21,600 | 1 день | 93.8% F1 |
| + TensorRT FP16 | 20 T4 | $12,000 | 3 дня | 93.8% F1 |
| + INT8 + batching | 8 T4 | $4,800 | 5 дней | 93.2% F1 |
| + DistilBERT | 5 T4 | $3,000 | 2 недели | 91.5% F1 |

**Расчёт:**
- T4 в облаке: ~$600/month (on-demand), ~$250/month (reserved)
- Стоимость разработчика: ~$100/hour

**ROI оптимизации:**

```
Экономия при переходе от Baseline к INT8 + batching:
- Инфра: $36,000 - $4,800 = $31,200/month
- Dev cost: 5 дней × 8 часов × $100 = $4,000 (единоразово)
- Payback period: $4,000 / $31,200 = 0.13 месяца (~4 дня)

Экономия: $31,200 × 12 - $4,000 = $370,400/year
```

## 7.7 Рекомендуемый подход

На основе анализа компромиссов для рассматриваемого сценария рекомендуется:

### Обязательно применять:
1. **FP16 Mixed Precision** — нет негативных последствий
2. **TensorRT** — значительное ускорение при приемлемой сложности
3. **Dynamic Batching** — критично для достижения throughput целей

### Рекомендуется применять:
4. **INT8 Quantization** — если допустима деградация 0.5-1%
5. **FlashAttention** — при обработке последовательностей >128 токенов

### Применять при необходимости:
6. **DistilBERT** — только если другие методы недостаточны и допустима деградация 2-3%
7. **Head Pruning** — как дополнительная оптимизация с marginal benefit

### Не рекомендуется:
8. **Unstructured Pruning** — нет аппаратной поддержки на T4
9. **Early Exit** — слишком сложно для marginal gain

## 7.8 Резюме

Ключевые принципы при выборе оптимизаций:

1. **Start simple:** Начните с FP16 и TensorRT — минимальные риски, значительный эффект
2. **Measure first:** Профилируйте перед оптимизацией, чтобы подтвердить bottlenecks
3. **Validate quality:** Каждая оптимизация должна проходить regression testing
4. **Consider TCO:** Учитывайте не только скорость, но и стоимость разработки и поддержки
5. **Plan for rollback:** Всегда иметь возможность откатиться к предыдущей версии
