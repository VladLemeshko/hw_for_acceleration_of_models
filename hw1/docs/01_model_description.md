# 1. Описание модели и контекста применения

## 1.1 Выбранная модель

Для анализа выбрана модель **BERT-base-uncased** (Bidirectional Encoder Representations from Transformers) — предобученная языковая модель, разработанная исследователями Google в 2018 году. BERT стала одной из наиболее влиятельных архитектур в области обработки естественного языка (NLP) и до сих пор активно применяется в продуктовых системах.

### Основные характеристики модели

| Параметр | Значение |
|----------|----------|
| Архитектура | Transformer Encoder |
| Количество слоёв (L) | 12 |
| Размер скрытого состояния (H) | 768 |
| Количество голов attention (A) | 12 |
| Общее число параметров | ~110 млн |
| Максимальная длина последовательности | 512 токенов |
| Словарь | WordPiece, 30 522 токена |

## 1.2 Решаемая задача

Модель применяется для **многоклассовой классификации текстовых обращений** пользователей в системе технической поддержки крупного SaaS-продукта. Входящие текстовые запросы необходимо автоматически категоризировать по одной из 50 тематических категорий для последующей маршрутизации к соответствующим специалистам.

### Примеры категорий обращений

- Проблемы с авторизацией
- Вопросы по биллингу
- Технические сбои
- Запросы на функциональность
- Жалобы на качество сервиса

### Формат входных данных

Входом модели является текстовое обращение пользователя длиной от 10 до 500 символов (в среднем ~150 символов, что соответствует ~40-60 токенам после токенизации). Тексты на русском и английском языках в соотношении примерно 70/30.

## 1.3 Сценарий использования

### Режим работы: онлайн-инференс

Модель работает в режиме **реального времени** (online inference), обрабатывая входящие запросы по мере их поступления. Это накладывает жёсткие требования на латентность ответа, поскольку пользователь ожидает немедленной реакции системы.

### Характеристики нагрузки

| Метрика | Значение | Комментарий |
|---------|----------|-------------|
| Средняя частота запросов | 500 RPS | Типичная дневная нагрузка |
| Пиковая частота запросов | 2000 RPS | Пиковые часы (10:00-12:00, 14:00-17:00) |
| Средний размер запроса | 60 токенов | После токенизации |
| Максимальный размер запроса | 256 токенов | Ограничение на уровне API |

### Требования к производительности

| Требование | Значение | Приоритет |
|------------|----------|-----------|
| Латентность (p50) | ≤ 20 мс | Высокий |
| Латентность (p99) | ≤ 50 мс | Критический |
| Пропускная способность | ≥ 1000 RPS | Высокий |
| Доступность | 99.9% | Критический |

### Обоснование требований к латентности

Требование латентности p99 ≤ 50 мс обусловлено следующими факторами:

1. **Пользовательский опыт**: Классификация является частью pipeline обработки обращения. Суммарное время ответа системы не должно превышать 200 мс, из которых на ML-компонент выделяется не более 50 мс.

2. **SLA перед клиентами**: Договорные обязательства перед корпоративными клиентами фиксируют максимальное время первичной обработки обращения.

3. **Масштабируемость**: При пиковой нагрузке в 2000 RPS увеличение латентности приводит к накоплению очереди запросов и деградации всей системы.

## 1.4 Бизнес-контекст

### Текущее состояние

В настоящий момент модель развёрнута на кластере из 4 серверов с GPU NVIDIA T4. При текущей конфигурации система обеспечивает требуемую производительность, однако с ростом клиентской базы ожидается увеличение нагрузки на 50-100% в течение следующего года.

### Цели оптимизации

1. **Снижение инфраструктурных затрат**: Оптимизация модели позволит обслуживать возросшую нагрузку без пропорционального увеличения количества серверов.

2. **Улучшение латентности**: Снижение p99 латентности с 50 мс до 30 мс повысит качество пользовательского опыта.

3. **Подготовка к edge-развёртыванию**: В перспективе рассматривается развёртывание легковесной версии модели непосредственно в мобильных приложениях для офлайн-классификации.

## 1.5 Метрики качества

### Текущие показатели модели

| Метрика | Значение на тестовой выборке |
|---------|------------------------------|
| Accuracy | 94.2% |
| Macro F1-score | 89.7% |
| Weighted F1-score | 93.8% |

### Допустимая деградация качества

При оптимизации модели допускается снижение качества не более чем на **2-3 процентных пункта** по метрике Weighted F1-score, что соответствует минимально допустимому значению ~91%.
