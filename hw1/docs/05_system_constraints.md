# 5. Системные ограничения

## 5.1 Обзор ограничений

Системные ограничения определяют границы, в которых должна функционировать модель BERT в рассматриваемом сценарии. Эти ограничения формируются требованиями бизнеса, техническими возможностями платформы и операционными соображениями.

## 5.2 Классификация ограничений

### По критичности

| Уровень | Описание | Последствия нарушения |
|---------|----------|----------------------|
| **Критичные** | Обязательные к выполнению | Отказ системы, нарушение SLA |
| **Высокой важности** | Желательные к выполнению | Деградация качества сервиса |
| **Средней важности** | Предпочтительные | Повышенные затраты |
| **Низкой важности** | Желательные улучшения | Упущенные возможности |

## 5.3 Ограничения по латентности

### 5.3.1 Требования

| Метрика | Значение | Критичность |
|---------|----------|-------------|
| p50 латентность | ≤ 20 мс | Высокая |
| p95 латентность | ≤ 35 мс | Высокая |
| **p99 латентность** | **≤ 50 мс** | **Критическая** |
| p99.9 латентность | ≤ 100 мс | Средняя |

### 5.3.2 Обоснование

**Почему p99 ≤ 50 мс критично:**

1. **SLA с клиентами:** Корпоративные договоры фиксируют максимальное время обработки обращения. ML-компонент — часть этого pipeline.

2. **Накопление задержек:** При превышении латентности образуется очередь запросов:
   ```
   При 2000 RPS и латентности 60 мс (вместо 50 мс):
   Доп. нагрузка = 2000 × 0.01 × 10 мс = 200 мс/с накапливаемой задержки
   За 10 секунд: очередь из 2000 запросов
   ```

3. **Пользовательский опыт:** Исследования показывают, что задержка >100 мс воспринимается пользователем как "тормозящая" система.

## 5.4 Ограничения по пропускной способности

### 5.4.1 Требования

| Метрика | Значение | Критичность |
|---------|----------|-------------|
| Средняя пропускная способность | ≥ 500 RPS | Высокая |
| **Пиковая пропускная способность** | **≥ 2000 RPS** | **Критическая** |
| Устойчивая пиковая нагрузка | ≥ 1000 RPS (30 мин) | Высокая |

### 5.4.2 Расчёт требуемых ресурсов

При целевой латентности 30 мс и пропускной способности 2000 RPS:

**Последовательная обработка (batch=1):**
```
1 GPU: 1000 мс / 30 мс = 33 RPS
Для 2000 RPS: 2000 / 33 = 61 GPU
```

**С динамическим батчингом (batch=8, latency=40 мс):**
```
1 GPU: 8 × (1000 / 40) = 200 RPS
Для 2000 RPS: 2000 / 200 = 10 GPU
```

**Вывод:** Без оптимизаций требуется 10-60 GPU для обслуживания пиковой нагрузки, что экономически нецелесообразно.

## 5.5 Ограничения по памяти

### 5.5.1 Память GPU

| Ресурс | Доступно | Используется | Запас |
|--------|----------|--------------|-------|
| VRAM | 16 GB | ~3-4 GB | 12-13 GB |
| Model weights (FP16) | — | 220 MB | — |
| Activations (batch=32) | — | ~500 MB | — |
| Framework overhead | — | ~2 GB | — |
| KV cache (если применимо) | — | N/A | — |

**Критичность:** Средняя. Текущая конфигурация не исчерпывает память GPU.

### 5.5.2 Память CPU/RAM

| Ресурс | Доступно | Используется | Запас |
|--------|----------|--------------|-------|
| System RAM | 128 GB | ~16 GB | 112 GB |
| Model copies | — | ~2 GB | — |
| Tokenizer cache | — | ~500 MB | — |
| Request buffers | — | ~2 GB | — |

**Критичность:** Низкая. Память не является ограничивающим фактором.

### 5.5.3 Ограничения при масштабировании

При увеличении batch size память GPU может стать ограничением:

```
Память активаций ≈ batch × seq_len × hidden × num_layers × 4 bytes
При batch=64, seq=256: 64 × 256 × 768 × 12 × 4 ≈ 600 MB
При batch=128, seq=256: ~1.2 GB
При batch=256, seq=256: ~2.4 GB
```

## 5.7 Ограничения по качеству модели

### 5.7.1 Требования к качеству

| Метрика | Текущее значение | Минимально допустимое | Критичность |
|---------|------------------|----------------------|-------------|
| **Weighted F1** | **93.8%** | **≥91%** | **Критическая** |
| Accuracy | 94.2% | ≥92% | Высокая |
| Macro F1 | 89.7% | ≥87% | Средняя |

### 5.7.2 Допустимая деградация

При оптимизации модели допускается снижение качества не более чем на **2-3 п.п.** по основной метрике (Weighted F1).

**Обоснование:**
- Снижение F1 ниже 91% приводит к заметному росту ошибок маршрутизации
- Каждый процент ошибок = ~500 неправильно маршрутизированных обращений в день
- Цена ошибки маршрутизации: увеличение времени решения на ~30 минут

## 5.11 Резюме критических ограничений

Для выбранного сценария использования **критическими** являются следующие ограничения:

1. **Латентность p99 ≤ 50 мс** — определяет возможность использования в онлайн-режиме
2. **Пропускная способность ≥ 2000 RPS** — определяет способность обслуживать пиковую нагрузку
3. **Качество Weighted F1 ≥ 91%** — определяет бизнес-ценность модели
4. **Доступность ≥ 99.9%** — определяет надёжность сервиса

Эти ограничения формируют пространство допустимых решений при оптимизации модели: любая оптимизация должна сохранять соответствие критическим ограничениям.
